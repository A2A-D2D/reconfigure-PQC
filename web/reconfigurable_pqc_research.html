SPUV3 RCE + FE 阵列研究与竞品分析

基于 SPUV3 RCE 的扩展定位

这次设计不是另起一套独立处理器，而是在 SPUV3 RCE Reference Guide 描述的系统上补一条 Falcon FE 路径。 SPUV3 已有 RV64IMFD 标量核、BNPU、VPU、RSA、ILM/DLM/DPRAM、AHB 配置桥和 XPqc 工具链；FE 阵列应作为 VPU/PQC 扩展接入。

Baseline

现有 RCE 能力

RV64IMFD 负责控制，BNPU 覆盖大数/国密/传统模运算，VPU 覆盖 Kyber/Dilithium 的 320-bit 向量 NTT，RSA accelerator 通过 DPRAM 协同。

Gap

Falcon 的缺口

Falcon 签名和密钥生成依赖 f64/fpr 复数 FFT、iFFT、twiddle 乘法和 tree 相关数值计算，单靠标量 FPU 会让指令数和搬运压力过高。

Extension

FE 阵列的职责

FE 作为 VPU 旁路或子执行单元，接收 XPqc/CSR 配置，以 task 粒度执行 CT/GS butterfly、complex mul、complex add/sub 和局部 in-place 写回。

Interface

和 SPUV3 接口如何对齐

控制侧沿用 XPqc/SFR/CSR；状态侧并入 `spuv3_busy`、task done 和 ordered commit；数据侧从 VR/DLM/DPRAM 进入 FE buffer，再通过 lane tag 写回输出 slice。

Width

320-bit VR 与 FE lane

SPUV3 VR 是 320-bit，天然对应 5 个 f64 lane；当前 FE 原型常用 8 lane/512-bit。集成时可以选择 `LANES=5`、两拍 VR 搬运，或增加本地 FE buffer。

为什么研究可重构 PQC

NIST 已发布 ML-KEM、ML-DSA、SLH-DSA 三个主要 PQC FIPS 标准，并继续推进 Falcon、HQC 等后续标准化方向。 硬件侧的问题是：这些算法不是同一种计算模式。只做一个 NTT 加速器很快，但算法覆盖范围有限；做通用处理器灵活，但能效不足。

Algorithm Diversity

算法数学问题不同

ML-KEM/ML-DSA 以模格和 NTT 为核心，Falcon 还需要双精度复数 FFT，SLH-DSA 则偏哈希密集型。统一硬件要能覆盖多种数据宽度和运算域。

Crypto Agility

标准和部署会变化

企业迁移周期长，算法组合会随标准、风险和应用约束变化。可重构处理器比固定 ASIC 更容易适配新 profile 和备选算法。

Data Movement

数据搬运常常成为瓶颈

NTT/FFT 每一层都要读写向量和 twiddle。高性能设计不只是堆乘法器，还要处理 bank 冲突、in-place 写回、转置和 forward。

当前研究路线

公开研究大致可以分为四类：单算法加速器、共享 NTT 加速器、FFT/NTT 统一加速器、以及面向多数学问题的可重构 PQC 处理器。

Route 1

固定算法加速器

针对 Kyber、Dilithium、Falcon 或 SPHINCS+ 某一个算法做深度优化。优势是面积/时序容易收敛，缺点是迁移和复用能力弱。

Route 2

共享 NTT 核

围绕 ML-KEM、ML-DSA 等 lattice 算法复用 NTT、模乘、采样和 hash datapath。覆盖主流标准较好，但 Falcon 的 FFT/fpr 路径仍然是缺口。

Route 3

FFT/NTT 统一加速器

试图复用复数 FFT 和模域 NTT 的 butterfly、存储访问、排列网络和乘法资源。适合 Falcon 这类同时涉及 NTT/FFT 的签名算法。

Route 4

可重构 PQC 处理器

以 arithmetic cluster、task scheduler、banked buffer 和可配置 AE/FE 为中心，把多个算法映射成 coarse-grained tasks。这是最接近“PQC 领域处理器”的路线。

竞品与代表性方案

这里的“竞品”不只指商业产品，也包括论文级原型和可作为技术路线对照的公开架构。

类别 代表方案 优势 不足 对本项目的启发
多数学问题 PQC 处理器 PQPU、RePQC 类 agile processor 支持多类算法任务；强调 task clustering、banked buffer、AE/FE array 和 vector forwarding。 架构复杂，scheduler、存储、crossbar、任务格式都需要较完整的系统设计。 SPUV3 已有 VPU/BNPU/RSA 基础，FE 扩展采用 shared FE、lane_mask、in-place 输出聚合来补齐 Falcon 路径。
标准 lattice 加速器 Kyber / Dilithium NTT accelerator NTT、模乘、采样可以做到高吞吐和低面积；最贴近 ML-KEM/ML-DSA 主流部署。 多数设计不覆盖 Falcon 的 f64 FFT，算法 agility 有边界。 SPUV3 现有 VPU/NTT 路径应保持简洁高效，同时通过 FE 扩展补 Falcon FFT。
FFT/NTT 统一加速器 Unified FFT/NTT accelerator for Falcon 等研究 尝试复用 butterfly、存储排列和乘法资源，适合同时包含 NTT 与 FFT 的算法。 数值域差异大，模约减和 f64/fixed-point 精度路径很难完全统一。 SPUV3 保留 VPU 整数路径与 FE 浮点复数路径分工，在 wrapper/operator 层统一 CT/GS、lane、mask 和 task 接口。
FPGA / IP 型方案 NTT IP、Keccak IP、Falcon FFT IP 开发快，适合验证单个瓶颈模块。 跨算法集成和数据搬运常常不足，最终能效依赖系统拼接质量。 当前 RTL 可以作为 FE IP 原型，但最终目标是接入 SPUV3 VPU/XPqc 体系。
通用 CPU / SIMD / GPU AVX2/NEON/GPU PQC software 灵活，易更新，适合软件迁移和服务器环境。 能效、延迟确定性和嵌入式部署不如专用硬件。 硬件需要保留可配置性，但不能退化成通用浮点/整数执行。

本项目当前定位

当前项目更准确地说是 SPUV3 RCE 的 Falcon FE 阵列扩展原型：现有 VPU/AE 覆盖整数模域和 NTT，新增 FE 覆盖 Falcon f64/complex FFT 路径，operator 层把它们组织成 XPqc task。

VPU 现有 320-bit 向量、NTT、Kyber/Dilithium
FE 新增 f64、复数、FFT、CT/GS butterfly
3 reference / pipeline / shared 三种实现档位
mask 对齐 XPqc/CSR 的 lane 级 task 配置
Strength

已经具备的差异化点

FE 不是简单 FPU，而是 SPUV3 VPU/PQC 路径上的复数/FFT 执行阵列；shared FE 支持 `ready_in` 背压、`lane_mask`、输出寄存器直接聚合，适合低面积扩展。

Gap

距离完整处理器还缺什么

还需要 `spuv3_vpu_fe_wrap`、VR/DLM pack-unpack、twiddle cache、permutation network、XPqc 编码、真实 Falcon FFT/iFFT stage 测试，以及 ordered commit 互锁。

后续研究路线

P0

新增 SPUV3 FE Wrapper

把 FE operator 接入 XPqc/CSR/SFR，完成 VR/DLM pack-unpack、busy/done 和 ordered commit 互锁。

P1

LANES=5 与两拍搬运

评估 320-bit VR 对齐方案：直接 5 lane、保留 8 lane 两拍拼接，或增加本地 FE buffer。

P2

Permutation + twiddle cache

补齐 Falcon FFT/iFFT stage 的排列网络和 twiddle cache，降低 VR/DLM 带宽压力。
